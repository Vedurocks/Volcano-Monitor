3. DATA PAYLOAD STRUCTURE
═══════════════════════════════════════════════════════════════════════════════

A. INITIALIZATION PACKET (Sent once on Arduino connection)
───────────────────────────────────────────────────────────
Format:  MIN_ULTRA:[Value],MIN_VIB:[Value]
Example: MIN_ULTRA:10.0,MIN_VIB:3

Sent when:
- User clicks "Connect" button
- Successfully establishes serial connection to Arduino
- Only sent ONCE per connection session

B. LIVE SENSOR PACKET (Sent every 2-3 seconds in loop)
───────────────────────────────────────────────────────────
Format: [Distance],[Vibration],[Dist_Alert],[Vib_Alert],[Dual_Alert],[Kill],[Sys],[MinU],[MinV]

Status Constants:
- Dist_Alert:  "DIST_OK" or "DIST_ALERT"
- Vib_Alert:   "VIB_OK" or "VIB_ALERT"
- Dual_Alert:  "DUAL_OK" or "DUAL_ALERT"
- Kill:        "KILL_OFF" or "KILL_ON"
- Sys:         "SYS_OFF" or "SYS_ON"

Example Live Packets:
45.67,0,DIST_OK,VIB_OK,DUAL_OK,KILL_OFF,SYS_ON,10.0,3
43.21,1,DIST_OK,VIB_OK,DUAL_OK,KILL_OFF,SYS_ON,10.0,3
8.50,5,DIST_ALERT,VIB_ALERT,DUAL_ALERT,KILL_OFF,SYS_ON,10.0,3
8.30,4,DIST_ALERT,VIB_ALERT,DUAL_ALERT,KILL_ON,SYS_ON,10.0,3

FIELD BREAKDOWN:
1. Distance      - 45.67       (float, 2 decimals, cm)
2. Vibration     - 512         (integer, 0-1023, raw ADC)
3. Dist_Alert    - DIST_OK     (string status)
4. Vib_Alert     - VIB_OK      (string status)
5. Dual_Alert    - DUAL_OK     (string status)
6. Kill          - KILL_OFF    (string status)
7. Sys           - SYS_ON      (string status)
8. MinU          - 10.0        (float, 1 decimal, cm)
9. MinV          - 3           (integer)

4. EXE LOGIC IMPLEMENTATION
═══════════════════════════════════════════════════════════════════════════════

✓ REQUIREMENT 1: Persistent sensor reading loop
  - App continuously reads from serial port at ~500ms intervals
  - Arduino sends: distance,seismic,dist_alert,seis_alert,dual_alert,kill,sys,min_ultra,min_seis
  - Data parsed and displayed in real-time UI

✓ REQUIREMENT 2: Non-blocking cloud sync
  - POST requests run in background threads
  - Failed requests do NOT hang the EXE
  - UI remains responsive during network issues
  - 3-second timeout per request prevents indefinite blocking

✓ REQUIREMENT 3: Local caching on failure
  - Failed POST requests are cached in memory
  - Cache limit: 100 most recent failed requests
  - Old cached items (>5 minutes) are automatically discarded
  - Retry logic: Up to 5 cached requests retried per sync cycle

✓ REQUIREMENT 4: Throttling (2-3 seconds interval)
  - Arduino sends data every ~500ms
  - App BUFFERS latest data
  - Cloud sync sends every 2.5 seconds (configurable via CLOUD_SEND_INTERVAL)
  - Prevents API overload and excessive network traffic

5. IMPLEMENTATION DETAILS
═══════════════════════════════════════════════════════════════════════════════

CODE ARCHITECTURE:
┌──────────────────────────────────────────────────────────────┐
│ Arduino (500ms) → Serial → App                               │
│                             ↓                                │
│                     _parse() method                          │
│                             ↓                                │
│              _send_to_cloud() (buffer)                       │
│                             ↓                                │
│         Check throttle timer (2.5s)                          │
│                  ↓              ↓                            │
│            Too soon         Time elapsed                     │
│            (skip)         (start worker)                     │
│                                ↓                             │
│                        _cloud_worker()                       │
│                                ↓                             │
│                      _post_to_api()                          │
│                       ↓        ↓                             │
│                  Success    Failure                          │
│                      ↓          ↓                            │
│                   Done    Add to cache                       │
│                              ↓                                │
│                    Retry cached items                        │
└──────────────────────────────────────────────────────────────┘

KEY METHODS:

1. _send_to_cloud(dist, seis, dist_alert, ...)
   - Called from _parse() on EVERY Arduino packet (~500ms)
   - Buffers latest data in self._last_data
   - Checks throttle timer (self._last_cloud_send)
   - If >= 2.5s elapsed, spawns _cloud_worker() thread

2. _cloud_worker()
   - Background thread (non-blocking)
   - Formats buffered data into payload string
   - Calls _post_to_api() to send current data
   - If failed, adds to self._cloud_cache
   - Retries up to 5 cached failed requests
   - Discards cache entries older than 5 minutes

3. _post_to_api(payload)
   - Executes single HTTP POST request
   - 3-second timeout
   - Returns True on success (200-299 status)
   - Returns False on failure (network error, timeout, 4xx/5xx)

4. _send_startup_message()
   - Called from _connect() when Arduino connects
   - Sends MIN_ULTRA:X,MIN_VIB:Y format
   - Non-critical (silent failure if POST fails)

INSTANCE VARIABLES:

self._last_cloud_send = 0     # Timestamp of last successful send
self._cloud_cache = []        # List of failed request payloads
self._last_data = {}          # Buffer for latest sensor reading

CONSTANTS:

VOLCANO_API = "https://volcano-monitor-g4hl.vercel.app/api/volcano"
CLOUD_SEND_INTERVAL = 2.5     # Seconds between cloud updates
PRISMA_DB = "postgres://..."  # Direct DB connection (future use)

8. CONFIGURATION OPTIONS
═══════════════════════════════════════════════════════════════════════════════

CHANGE SYNC INTERVAL:
Edit constant at top of file:
    CLOUD_SEND_INTERVAL = 2.5  # Change to 3.0 for 3-second interval

CHANGE CACHE SIZE:
In _cloud_worker() method:
    if len(self._cloud_cache) > 100:  # Change 100 to desired max

CHANGE CACHE RETENTION:
In _cloud_worker() method:
    if time.time() - cached['timestamp'] > 300:  # 300 = 5 minutes

CHANGE TIMEOUT:
In _post_to_api() method:
    timeout=3  # Change to 5 for 5-second timeout

CHANGE RETRY COUNT:
In _cloud_worker() method:
    while self._cloud_cache and retry_count < 5:  # Change 5 to desired max

SWITCH TO DIRECT PRISMA DB (Advanced):
1. Install psycopg2: pip install psycopg2-binary
2. Replace _post_to_api() with direct INSERT using PRISMA_DB connection string
3. Requires SQL schema knowledge and error handling

9. TROUBLESHOOTING
═══════════════════════════════════════════════════════════════════════════════

Q: Data not appearing in Vercel?
A: 1. Enable debug logging (see section 7)
   2. Check internet connection
   3. Verify API URL is correct
   4. Check Vercel deployment logs
   5. Test API with curl: curl -X POST -d "test" https://volcano-monitor-g4hl...

Q: App freezing when connected?
A: Should NOT happen - cloud sync is fully async
   If it does: check for exceptions in _cloud_worker thread

Q: Too many API requests?
A: Increase CLOUD_SEND_INTERVAL (e.g., 3.0 or 4.0)

Q: Not enough API requests?
A: Decrease CLOUD_SEND_INTERVAL (minimum 1.0 recommended)

Q: Cache growing too large?
A: 1. Check network connection
   2. Lower max cache size
   3. Reduce cache retention time

Q: Old data being sent?
A: Cache retention is 5 minutes - this is by design
   Reduce retention time if needed

- Negligible impact on app performance
