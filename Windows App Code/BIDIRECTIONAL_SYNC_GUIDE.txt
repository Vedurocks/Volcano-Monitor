VOLCANO ERUPTION MONITOR v5.0 — BIDIRECTIONAL CLOUD SYNC

═══════════════════════════════════════════════════════════════════════════════
OVERVIEW
═══════════════════════════════════════════════════════════════════════════════

The app implements a two-way cloud synchronization system:

1. **UPLINK** (App → Cloud):
   - Real-time sensor data from Arduino
   - Posted as JSON immediately on every sensor reading (~500ms)

2. **DOWNLINK** (Cloud → App):
   - Remote commands from database
   - Polled every 1 second via GET request
   - Executed on Arduino automatically

═══════════════════════════════════════════════════════════════════════════════
API ENDPOINT
═══════════════════════════════════════════════════════════════════════════════

URL: https://volcano-monitor-g4hl.vercel.app/api/volcano

METHODS:
- POST: Send sensor data OR mark command as executed
- GET:  Retrieve pending commands

═══════════════════════════════════════════════════════════════════════════════
1. UPLINK: SENSOR DATA (App → Cloud)
═══════════════════════════════════════════════════════════════════════════════

ARDUINO SERIAL FORMAT (CSV, 9600 baud):
distance,vibration,dist_alert,vib_alert,dual_alert,kill_state,sys_state,min_ultra,min_vib\n

EXAMPLE:
243.5,312,0,0,0,0,1,100,50

FIELD DETAILS:
- distance:    243.5  (float, cm)
- vibration:   312    (int, 0-1023, raw ADC)
- dist_alert:  0      (0 or 1)
- vib_alert:   0      (0 or 1)
- dual_alert:  0      (0 or 1)
- kill_state:  0      (0 or 1)
- sys_state:   1      (0 or 1)
- min_ultra:   100    (int, cm)
- min_vib:     50     (int)

APP CONVERTS TO JSON AND POSTS:

POST https://volcano-monitor-g4hl.vercel.app/api/volcano
Content-Type: application/json

{
  "distance": 243.5,
  "vibration": 312,
  "dist_alert": false,
  "vib_alert": false,
  "dual_alert": false,
  "kill_state": false,
  "sys_state": true,
  "min_ultra": 100,
  "min_vib": 50
}

TIMING:
- Sent immediately on every Arduino packet (no throttling)
- Frequency: ~120 requests/minute (every ~500ms)
- Non-blocking: POST runs in background thread
- Timeout: 3 seconds per request
- Failures: Silent (won't interrupt app)

═══════════════════════════════════════════════════════════════════════════════
2. DOWNLINK: COMMAND POLLING (Cloud → App)
═══════════════════════════════════════════════════════════════════════════════

GET REQUEST (every 1 second):

GET https://volcano-monitor-g4hl.vercel.app/api/volcano

EXPECTED RESPONSE:

{
  "data": {
    "distance": 243.5,
    "vibration": 312,
    ...
  },
  "commands": [
    {"id": 12, "command": "on"},
    {"id": 13, "command": "min ult 150"}
  ]
}

COMMAND PROCESSING:

1. App receives commands array from GET response
2. For each command object:
   a. Extract id and command string
   b. Send command to Arduino via serial
   c. Log to UI (terminal and shell)
   d. Immediately POST executed_id to mark as done

EXAMPLE COMMAND EXECUTION FLOW:

T=0.0s  GET /api/volcano
        → Response: {"commands": [{"id": 12, "command": "on"}]}

T=0.01s Send "on\n" to Arduino via serial
        → UI log: "☁ Remote command: on"

T=0.02s POST executed confirmation:
        {
          "executed_id": 12
        }

T=1.0s  GET /api/volcano (next poll)
        → Response: {"commands": []} (command 12 now marked executed)

SUPPORTED COMMANDS:
- on
- off
- kill
- reset
- 456
- min ult X  (e.g., "min ult 150")
- min sei X  (e.g., "min sei 75")

═══════════════════════════════════════════════════════════════════════════════
3. EXECUTED CONFIRMATION (App → Cloud)
═══════════════════════════════════════════════════════════════════════════════

After executing a command, immediately POST:

POST https://volcano-monitor-g4hl.vercel.app/api/volcano
Content-Type: application/json

{
  "executed_id": 12
}

This tells the database:
- Command with id=12 has been executed
- Don't return it in future GET requests
- Update command status to "executed"

4. IMPLEMENTATION DETAILS
═══════════════════════════════════════════════════════════════════════════════

CODE ARCHITECTURE:

┌────────────────────────────────────────────────────────────────┐
│                      UPLINK (Sensor Data)                      │
├────────────────────────────────────────────────────────────────┤
│ Arduino (500ms) → Serial → _parse()                            │
│                              ↓                                 │
│                    _send_to_cloud()                            │
│                              ↓                                 │
│                 POST JSON (background thread)                  │
│                              ↓                                 │
│                    Vercel API Database                         │
└────────────────────────────────────────────────────────────────┘

┌────────────────────────────────────────────────────────────────┐
│                   DOWNLINK (Remote Commands)                   │
├────────────────────────────────────────────────────────────────┤
│ Vercel API Database                                            │
│         ↓                                                      │
│ GET every 1s (_command_poll_loop)                             │
│         ↓                                                      │
│ Parse "commands" array                                         │
│         ↓                                                      │
│ For each command:                                              │
│   1. _execute_remote_command()                                 │
│   2. Send to Arduino via serial                                │
│   3. Log to UI                                                 │
│   4. POST {"executed_id": X}                                   │
└────────────────────────────────────────────────────────────────┘

KEY METHODS:

_send_to_cloud(dist, seis, dist_alert, ...)
  - Called from _parse() on every Arduino packet
  - Builds JSON payload
  - POST in background thread (non-blocking)
  - 3-second timeout
  - Silent failures

_start_command_polling()
  - Called when Arduino connects
  - Spawns background thread running _command_poll_loop()
  - Sets self._command_poll_active = True

_stop_command_polling()
  - Called when Arduino disconnects
  - Sets self._command_poll_active = False
  - Loop exits on next iteration

_command_poll_loop()
  - Infinite loop while connected
  - GET /api/volcano every 1 second
  - Parse response for "commands" array
  - Call _execute_remote_command() for each
  - Sleep 1 second, repeat

_execute_remote_command(command, cmd_id)
  - Send command to Arduino via self._send()
  - Log to UI: "☁ Remote command: on"
  - POST {"executed_id": cmd_id} in background
  - Silent failures (command executed even if POST fails)

INSTANCE VARIABLES:

self._command_poll_active = False  # Polling loop state
self._last_command_poll = 0        # Timestamp (currently unused)

CONSTANTS:

VOLCANO_API = "https://volcano-monitor-g4hl.vercel.app/api/volcano"
COMMAND_POLL_INTERVAL = 1.0  # Seconds between GET requests

6. ERROR HANDLING
═══════════════════════════════════════════════════════════════════════════════

POST SENSOR DATA FAILURES:
- Network error → Silent fail, continue
- Timeout → Silent fail, continue
- 4xx/5xx status → Silent fail, continue
- App never hangs on failed POST

GET COMMAND FAILURES:
- Network error → Silent fail, retry in 1 second
- Timeout → Silent fail, retry in 1 second
- Invalid JSON → Silent fail, retry in 1 second
- Polling loop continues regardless

COMMAND EXECUTION FAILURES:
- Arduino not connected → Skip execution
- Serial error → Log error, continue polling
- executed_id POST fails → Silent fail (command still executed on Arduino)

CRITICAL: All network operations use try-except with silent failures.
The app NEVER hangs or crashes due to network issues.

8. DATABASE SCHEMA (SUGGESTED)
═══════════════════════════════════════════════════════════════════════════════

TABLE: sensor_data
- id (auto-increment)
- timestamp (datetime, auto)
- distance (float)
- vibration (int)
- dist_alert (boolean)
- vib_alert (boolean)
- dual_alert (boolean)
- kill_state (boolean)
- sys_state (boolean)
- min_ultra (int)
- min_vib (int)

TABLE: commands
- id (auto-increment)
- command (string, e.g., "on", "off", "min ult 150")
- created_at (datetime, auto)
- executed_at (datetime, nullable)
- status (enum: "pending", "executed")

API ENDPOINTS:

POST /api/volcano with JSON sensor data:
→ INSERT INTO sensor_data (...)

POST /api/volcano with {"executed_id": X}:
→ UPDATE commands SET status='executed', executed_at=NOW() WHERE id=X

GET /api/volcano:
→ SELECT * FROM sensor_data ORDER BY id DESC LIMIT 1
→ SELECT * FROM commands WHERE status='pending' ORDER BY id ASC
→ RETURN {"data": {...}, "commands": [...]}

9. PERFORMANCE METRICS
═══════════════════════════════════════════════════════════════════════════════

UPLINK TRAFFIC:
- Frequency: 120 POST requests/minute
- Payload size: ~150 bytes JSON per request
- Bandwidth: ~18 KB/minute outbound
- Database inserts: 120/minute (consider batching if too high)

DOWNLINK TRAFFIC:
- Frequency: 60 GET requests/minute
- Response size: ~200 bytes (no commands) or ~300 bytes (with commands)
- Bandwidth: ~12-18 KB/minute inbound
- Database queries: 60/minute

TOTAL BANDWIDTH: ~30-36 KB/minute (~2 MB/hour)

OPTIMIZATION OPTIONS (if needed):
1. Batch sensor data (send every 5s with array of readings)
2. Reduce GET polling to 2-3 seconds
3. WebSocket connection instead of polling (advanced)
